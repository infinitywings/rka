## Overview

This document is a **complete, annotated walkthrough** of a single RKA v2.1 research cycle, using a realistic systems research project as the example. Each step shows what the PI sees, what the agents do, and how the system state evolves.

The project demonstrates the full v2.1 lifecycle: from a one-sentence research idea through literature investigation, research design, iterative building, critique, and back to the next research question.

**Key v2.1 agents:**
- **Executor** — implementation, experimentation, data gathering (OpenClaw workspace-executor)
- **Researcher Brain** — synthesis, hypothesis generation, literature interpretation (OpenClaw workspace-researcher)
- **Reviewer Brain** — independent critique, quality gate (OpenClaw workspace-reviewer)
- **PI (you)** — human oversight, escalation resolution, strategic direction

## Phase 0 — Topic Germination

### What the PI does

The PI (Dr. Fu) starts with a rough research idea. In RKA, this is recorded as a **directive** note:

```
POST /api/notes
{
  "type": "directive",
  "source": "pi",
  "content": "Can federated learning be made practical on resource-constrained IoT devices without centralized training data? Investigate whether on-device gradient compression and adaptive synchronization intervals can reduce communication overhead enough to make FL viable on ARM Cortex-M class hardware (~256KB RAM)."
}
```

This directive becomes the **research charter** for the project. The PI sets the autonomy mode to `autonomous` once the direction is clear, allowing the agents to operate without constant PI intervention.

### What the system does

The directive is stored in the project journal. The research direction is recorded in the PI's role state. No agents are activated yet — this is just the initialization.

Privacy-utility frontier when combining FL with differential privacy on IoT data"

```

Emits `synthesis.created` → Reviewer picks it up.

### 1.5: Reviewer Brain critiques

Reviewer loads the synthesis, reads the literature entries, writes critique:

```
POST /api/notes
{
  "type": "note",
  "source": "reviewer",
  "content": "## Critique: Literature Synthesis\n\n**Verdict:** Sound structure, well-grounded in evidence. Gap identification is valid.\n\n**Concerns:**\n1. ```

Emits `disagreement.detected` → PI notified via session injection + WhatsApp.

### 1.6: PI resolves

PI reviews the escalation, adds the missing citation and clarifies Open Problem 2. Researcher revises synthesis. Reviewer approves with `critique.no_issues` → Researcher creates next mission.

## Phase 2 — Key Literature & Open-Source Project Selection

### 2.1: Executor evaluates frameworks

Mission: Evaluate Flower (https://github.com/adap/flower), PySyft (https://github.com/OpenMined/PySyft), and TensorFlow Federated (https://github.com/tensorflow/federated) for embedded FL use.

Executor uses `rka_browse_url` to visit GitHub READMEs, reads documentation, checks system requirements, evaluates:

```
Flower:
- Pros: Framework-agnostic (PyTorch, TensorFlow), active maintenance, good docs
- Memory: ~50MB base overhead (too high for Cortex-M, but server-side is fine)
- Embedded support: Yes, Flower mobile SDK for Android/iOS, embedded via Flask bridge

PySyft:
- Pros: Privacy tooling (PSI, secure aggregation), strong academic community
- Memory: Heavy (~500MB+), Python-native — unsuitable for embedded
- Embedded support: No

TensorFlow Federated:
- Pros: Google's production-grade, TFLite Micro integration is emerging
- Memory: ~30MB overhead (server), TFLite Micro on-device
- Embedded support: Best of the three for on-device

Decision recommendation: Flower (server) + custom C++ client on embedded. TFLite Micro as fallback."
```

Executor writes comparison note and creates a **decision** node in RKA:

```
POST /api/decisions
{
  "title": "Framework Selection: Flower + Custom C++ for FL on IoT",
  "decision": "Select Flower as the FL server framework. For embedded clients, build a lightweight C++ client using the Flower protocol (gRPC). TFLite Micro as secondary option for Android-based IoT gateways.",
  "rationale": "Flower offers best flexibility/overhead ratio. PySyft is too heavy. TFLite Micro is promising but less mature for FL protocol support.",
  "related_literature": ["<flower_paper_id>", "<tflite_micro_id>"],
  "alternatives_considered": ["PySyft (rejected — too heavy)", "TFF (rejected — too rigid for custom protocols)"]
}
```

### 2.2: Researcher reviews, Reviewer approves

Reviewer evaluates the framework selection, confirms Flower is the right choice given the embedded constraints. `critique.no_issues` emitted. Researcher acknowledges.

## Phase 3 — Research Design

### 3.1: PI directive

PI sets the research design direction:

```
POST /api/notes
{
  "type": "directive",
  "source": "pi",
  "content": "Design the full research. I expect: motivation paragraph, problem scope (what exactly, how exactly), 3 key challenges, proposed solution sketch, contribution list, novelty statement, system architecture overview, evaluation setup (datasets, baselines, metrics), and experimental methods. This should be a publishable research design."
}
```

Researcher creates a research design mission for Executor.

### 3.2: Executor drafts research design

Executor produces a comprehensive research design document:

**Motivation:**
> Federated learning (FL) enables training ML models across distributed devices without sharing raw data. However, current FL systems assume devices with sufficient compute and network capacity. IoT edge devices (ARM Cortex-M, ~256KB RAM) are left behind. This work addresses the forgotten 80% of IoT devices.

**Problem Scope:**
- Primary: Achieve FL training convergence on devices with <1MB RAM
- Secondary: Reduce per-round communication payload to <10KB
- Boundaries: Focus on image classification (CIFAR-10/16) and time-series (HAR); NLP out of scope

**Challenges:**
1. Memory: Model parameters alone (ResNet-8 ~1M params = 4MB) exceed device RAM
2. Bandwidth: Rural IoT uplink is 10-100× slower than typical FL assumptions
3. Heterogeneity: Non-IID data distribution across devices causes convergence instability

**Solution Sketch:**
- Gradient compression pipeline: top-k sparsification (1-5% nonzero gradients) + 8-bit quantization
- Adaptive sync: model staleness-aware weighting scheme (FedSA)
- Memory optimization: quantized model weights (int8), layer-by-layer gradient flushing

**Novelty:**
- First systematic study of FL viability on ARM Cortex-M class devices
- FedSA (Staleness-Aware Federated Averaging): 40% fewer rounds vs FedAvg on heterogeneous data
- Open-source benchmark suite for embedded FL (FLIoT-Bench)

**System Architecture:**
```
[IoT Devices] --(compressed gradients)--> [Flower Server] --(model updates)--> [Aggregated Model]
     |                                      |
  Cortex-M4F                           Ubuntu 20.04
  (256KB RAM)                          (GPU-accelerated)
  int8 weights                          FedSA aggregation
  layer-wise grad flush                  differential privacy
```

**Evaluation Setup:**
- Devices: Raspberry Pi Pico W (ARM Cortex-M33, 512KB RAM) + STM32F4 (Cortex-M4, 192KB RAM)
- Baselines: FedAvg, FedProx, SCAFFOLD
- Metrics: Accuracy (top-1), communication rounds to convergence, per-round bandwidth, memory peak, energy per round
- Datasets: CIFAR-10, HAR (Human Activity Recognition), anomaly detection (SMD dataset)
- SOTA comparison: Compare final accuracy and efficiency vs. published results on the same datasets

**Experimental Methods:**
- Ablation study on compression ratio vs. accuracy
- Heterogeneity analysis: Dirichlet(α) data splits with α ∈ {0.1, 0.5, 1.0}
- Energy measurement: INA219 power monitor on STM32

### 3.3: Researcher synthesizes → Reviewer critiques

Researcher synthesizes the design into a structured note, emits `synthesis.created`.

Reviewer critiques:
- Flags: "System architecture mentions 'differential privacy' but no parameters specified — should clarify σ, ε budget or remove claim"
- Flags: "Baseline SCAFFOLD requires 2× memory per device — how does this fit the 1MB constraint?"
- Approves with revisions note

Researcher revises, removes the unverified DP claim, adds a footnote about SCAFFOLD memory requirements.

`critique.no_issues` → Phase 3 complete.

## Phase 4 — Iterative Building & Results

### 4.1: Implementation Mission 1 — Baseline FL System

Executor picks up: "Build the Flower server + lightweight C++ client on Raspberry Pi Pico W."

Executor hits checkpoint:

```
Checkpoint submitted:
"Should we use gradient quantization (8-bit) or sparse compression (1% top-k) as the primary compression method? Quantization is simpler but provides only 4× reduction. Sparsification gives 100× but requires careful gradient tracking."
```

### 4.2: Researcher resolves checkpoint

Researcher reviews literature on gradient compression tradeoffs, recommends:

> "Use structured sparsity: top-k per-layer rather than global top-k. This preserves per-layer gradient importance while still achieving 20-50× compression. Layer-wise top-k with k=10% is a good starting point. Reference: Aji & Heafel, 'Sparse Gradient Compression for Distributed Learning' (ICLR 2017)."

Executor resumes, implements layer-wise top-k with 8-bit quantization fallback, completes v1 baseline system.

**Report submitted:**
- System works: achieves 91.2% accuracy on CIFAR-10 (vs 93.1% centralized — 1.9% gap)
- Compression ratio: 23× per round
- Memory peak: 847KB (exceeds 1MB budget — needs optimization)

### 4.3: Critique → Revision Mission

Reviewer: "847KB exceeds the stated scope. The gap between compressed model (4MB) and available RAM (1MB) needs to be addressed. Recommend quantized model weights (int8) + streaming layer processing to stay within 1MB."

Researcher creates follow-up mission: "Optimize memory footprint to <1MB. Use int8 quantized model (4-bit, grouped per layer), process gradients layer-by-layer to avoid holding full model in memory."

### 4.4: Iterative refinement

**Mission 2:** Implement int8 quantization + layer-wise gradient streaming.

- Round 1: Achieves 89.7% accuracy, 512KB memory — within budget ✅
- Round 2: Achieves 90.4% accuracy after tuning learning rate schedule ✅
- Round 3: Achieves 90.8% accuracy with improved aggregation weighting ✅

Each round: Executor reports → Researcher synthesizes → Reviewer critiques → Researcher creates next mission.

### 4.5: Ablation study mission

Researcher creates mission: "Run full ablation study — compression ratio (5%, 10%, 20%, 50%), dataset heterogeneity (α=0.1, 0.5, 1.0), device tiers (Cortex-M4 192KB vs Cortex-M33 512KB)."

Executor runs 36 experiments (4 compression × 3 heterogeneity × 3 repeats). Writes results tables.

### 4.6: Reviewer identifies inconsistency

Reviewer: "Experiment series D (α=0.1, compression=5%) shows inconsistent accuracy across runs: 87.2%, 89.1%, 88.3%. Standard deviation of 0.95% is unusually high — may indicate a measurement issue with the power meter affecting computation time, which affects gradient staleness."

`disagreement.detected` → PI notified.

PI resolves: "The INA219 power meter sampling rate was too low (100ms) causing timing jitter. Re-run D-series with 10ms sampling. The other series are unaffected."

Executor re-runs D-series. New results: 88.1%, 88.3%, 88.2% (σ=0.05%) — consistent. PI was right.

## Phase 5 — Evaluation & Refinement

After all iteration missions complete:

- Final accuracy: 91.4% on CIFAR-10 (vs 93.1% centralized, 1.7% gap)
- Memory footprint: 487KB peak (within 1MB budget ✅)
- Communication reduction: 23× per round ✅

Researcher synthesizes all results into a comprehensive evaluation note. Reviewer does final critique pass. `critique.no_issues` → next mission auto-created.
